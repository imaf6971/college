# Техническое задание для проекта "Система управления колледжем"
## 1. Цель проекта.

Цель проекта - спроектировать и разработать систему (далее просто **Система**) управления деятельностью студенческого совета в формате веб приложения.

Система позволит эффективно управлять направлениями студенческого совета, строить план мероприятий, просматривать рейтинг студентов строящийся на основе их участия в мероприятиях.

## 2. Функциональные блоки приложения.

Система состоит из следующих функциональных блоков:

1. Аутентификация и авторизация.
2. Управление деятельностью студенческого совета.
3. Ведение направлений студенческого совета.
4. Глобальный рейтинг студентов.

### 2.1 Типы пользователей

Система предусматривает два типа пользователей системы: Ольга Николаевна (Надо придумать чуть более официальное название для этой роли :) ) и руководитель направления студенческого совета. Ольга Николаевна назначает руководителей и следит за их деятельностью, а руководители направлений планируют и проводят мероприятия.

### 2.2 Регистрация

Система, как минимум в своей первой версии, не предполагает возможности для пользователей регистрироваться самостоятельно. Данные от аккаунта будут выдаваться единожды.

Данные от аккаунта будут представлять из себя:

* email
* пароль

### 2.3 Аутентификация и авторизация на основе системы ролей.

#### Описание.

**Аутентификация** позволяет обезопасить приложение от несанкционированного доступа, а **авторизация** позволит разделять функционал для различных пользователей на основе их **роли**.

#### 2.3.1 Аутентификация.

Любое действие пользователя, не прошедшего процедуру аутентификации (далее гостя) вызовет перенаправление на страницу аутентификации.

На данной странице от гостя потребуется ввести его данные от аккаунта.

Соответственно, гость не сможет получить доступ к системе, пока не выполнит процедуру аутентификации.

#### 2.3.2 Авторизация.

После выполнения **аутентификации**, пользователю будет доступен и виден только тот функционал, который соответствует его **роли**.

Попытка воспользоваться функционалом, не соответствующему роли пользователя, приведёт к появлению диалогового окна с сообщением: _"Доступ запрещён: недостаточно прав"_

### 2.4 Управление деятельностью студенческого совета.

#### Примечание

Данный функционал доступен роли ОН.

#### 2.4.1 Просмотр информации о существующих направлениях.

Информация о направлениях будет представлена таблицей со следующими колонками:

1. Название направления.
2. ФИО руководителя направления в формате _Миронов В. А._
3. Количество участников направления.

Так-же, в каждом ряду таблицы должна быть кнопка для переходу к просмотру информации о деятельности конкретного направления.

#### 2.4.2 Просмотр информации о деятельности направления.

Информация о направлении будет представлена страницей со следующим содержанием:

1. Название направления.
2. ***Подробная информация о руководителе направления.
3. ***Таблица с участниками направления.
4. ***Запланированные мероприятия.
5. ***Проведенные мероприятия.

### 2.5 Управление деятельностью направления.

#### Примечание

Данный функционал доступен роли руководитель направления.

### 2.5.1. Управление участниками направления

Включает в себя 
* Добавление участника.
* Удаление участника.

### 2.5.2 Управления мероприятиями

Данный функционал доступен роли руководитель направления.

Функционал:
1. Планирование мероприятий (Название мероприятия, дата проведения (от / до)).
2. Проведение мероприятия. Назначение ответственного, добавление участников с последующей оценкой качества их участия и заинтересованности.
3. Просмотр информации о проведённых мероприятиях.

### 4. Рейтинг студентов

Просмотр рейтинга групп, сформированный на основе участия студентов группы в мероприятиях.

Для реализации системы предлагается следующий стек технологий:

- СУБД **MySQL**

    - Язык программирования **Java**
    - Фреймворк **Spring Boot**
- Фронтенд
    - Движок шаблонов **Thymeleaf**
